# iCC 产品需求文档

## 定位

**小工具路线**：够用、够小、够稳、容易看懂。

不是功能丰富的 AI 平台，而是一个可靠、耐用、代码可读的微信桥接器。

## 核心特性

| 特性 | 说明 |
|------|------|
| 扫码接入 | 一条命令 `icc wechat setup`，扫码即用 |
| 对话记忆 | 通过 session resume 记住对话上下文 |
| 打字状态 | 处理期间显示"正在输入中" |
| 状态持久化 | 重启后自动恢复 session 和 token |

## 设计原则

1. **简单够用** — 不做多功能聚合，一个功能做透
2. **代码可读** — 新人接手能快速理解，文档少而精
3. **稳定优先** — 持久化、分层异常处理、优雅关闭
4. **体积小** — 核心代码 < 1000 行，无重型依赖

## 典型场景

- 用户在微信里问技术问题，Claude Code 在后台回答
- 开发者摸鱼时快速问代码片段
- 作为语音转文字的 AI 对话入口

## 用户故事

### US1: 扫码配置
作为用户，我想快速配置并开始使用。
- 运行 `icc wechat setup`
- 扫码登录，token 自动保存
- 运行 `icc start`，服务启动

### US2: 多轮对话
作为用户，我想在一个话题里继续对话。
- 首次消息创建 session
- 后续消息通过 `--resume` 恢复同一对话
- 每用户独立 session，互不干扰

### US3: 打字指示
作为用户，我想看到 Claude 正在处理。
- 发送消息后，微信显示"正在输入中"
- 处理完成后自动停止
- 失败不影响主流程

### US4: 重启恢复
作为用户，重启服务不想丢失对话状态。
- Session、token、游标持久化到磁盘
- 重启后自动加载
- 用户感受：重启前后是无缝的

## 非目标（不会做）

- 多 agent / 多 project
- 复杂的权限系统
- 对话导出 / AI memory
- Web 管理界面

## 质量要求

- 所有持久化操作有错误日志
- 单元测试覆盖核心逻辑
- 日志输出清晰可追溯
- 服务 crash 时 LaunchAgent 自动重启
